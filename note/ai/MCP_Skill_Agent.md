# MCP、Skill 和 Agent

> 在 AI 应用开发中，MCP、Skill 和 Agent 是三个常被提及的概念。它们分别代表了不同的抽象层次：MCP 是连接协议，Skill 是能力模块，Agent 是自主系统。理解三者的定位和关系，对于设计 AI 应用架构至关重要。

---

## 一、MCP（Model Context Protocol）

### 1.1 是什么

MCP（Model Context Protocol，模型上下文协议）是由 Anthropic 于 2024 年开源推出的一种开放标准协议，用于将 AI 助手与各类数据系统（数据库、文件系统、API 等）进行对接。

从技术定位来看，MCP 是一个**连接层协议**，它定义了 AI 模型与外部工具/数据源之间的通信规范。开发者可以通过 MCP 在数据源与 AI 工具之间构建安全的双向连接，使 AI 能够在多个工具和数据集之间保持上下文连贯。

MCP 的核心组件包括：

- **协议规范与 SDK**：定义通信标准和开发工具包
- **本地服务器支持**：如 Claude 桌面应用可连接本地 MCP 服务器
- **开源 MCP 服务器仓库**：提供预置的服务器实现

### 1.2 MCP 与 LLM Tool Call / Function Call 的关系

MCP 可以理解为 **LLM Tool Call 的标准化和扩展**：

| 特性           | Function Call / Tool Call              | MCP                               |
| -------------- | -------------------------------------- | --------------------------------- |
| **定义方式**   | 硬编码在应用代码中，需要开发者手动定义 | 标准化协议，动态发现可用工具      |
| **工具数量**   | 有限，受限于应用预定义的工具集         | 理论上无限，可通过 MCP 服务器扩展 |
| **上下文共享** | 每个工具独立，需要手动传递上下文       | 统一上下文管理，跨工具共享状态    |
| **连接方式**   | 直接 API 调用                          | 标准化协议，支持本地/远程连接     |

**关键区别**：

- Function Call 是 LLM 调用预定义函数的机制，MCP 是让 LLM **动态发现和调用**外部工具的协议
- MCP 将 Function Call 的概念**标准化**，让任何符合 MCP 规范的服务都能被 AI 模型调用
- MCP 提供了**工具发现机制**，AI 可以查询可用工具列表，而不需要硬编码

### 1.3 代码示例

#### 基础 MCP 服务器示例

```typescript
// server.ts - 一个简单的 MCP 服务器
import { Server } from "@modelcontextprotocol/sdk/server/index.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import {
  ListToolsRequestSchema,
  CallToolRequestSchema,
} from "@modelcontextprotocol/sdk/types.js";

const server = new Server(
  {
    name: "example-mcp-server",
    version: "1.0.0",
  },
  {
    capabilities: {
      tools: {},
    },
  },
);

// 注册可用工具
server.setRequestHandler(ListToolsRequestSchema, async () => ({
  tools: [
    {
      name: "read_file",
      description: "读取文件内容",
      inputSchema: {
        type: "object",
        properties: {
          path: {
            type: "string",
            description: "文件路径",
          },
        },
        required: ["path"],
      },
    },
    {
      name: "search_files",
      description: "搜索文件",
      inputSchema: {
        type: "object",
        properties: {
          pattern: {
            type: "string",
            description: "搜索模式",
          },
        },
        required: ["pattern"],
      },
    },
  ],
}));

// 处理工具调用
server.setRequestHandler(CallToolRequestSchema, async (request) => {
  const { name, arguments: args } = request.params;

  if (name === "read_file") {
    const content = await fs.readFile(args.path, "utf-8");
    return {
      content: [
        {
          type: "text",
          text: content,
        },
      ],
    };
  }

  if (name === "search_files") {
    const files = await glob(args.pattern);
    return {
      content: [
        {
          type: "text",
          text: JSON.stringify(files, null, 2),
        },
      ],
    };
  }

  return {
    content: [
      {
        type: "text",
        text: `Unknown tool: ${name}`,
      },
    ],
  };
});

const transport = new StdioServerTransport();
await server.connect(transport);
```

### 1.4 为什么选择这项技术

在 MCP 出现之前，AI 与外部系统的集成方式是高度碎片化的。每个 AI 应用需要为不同的数据源编写自定义连接器，导致：

- 集成成本高：每个系统都需要独立的适配代码
- 维护困难：连接器数量随工具增长呈线性膨胀
- 上下文割裂：AI 难以在多个工具间保持一致的上下文理解

MCP 的价值在于**以单一标准取代碎片化的集成方式**。通过统一的协议规范，开发者只需实现一次 MCP 接口，即可让 AI 连接到任何支持 MCP 的系统，大幅降低自定义连接器的开发和维护成本。

### 1.5 应用场景

MCP 的典型应用场景包括：

- **文件系统访问**：让 AI 能够读取、搜索和操作本地文件
- **数据库查询**：连接 PostgreSQL 等数据库，执行 SQL 查询并分析数据
- **版本控制集成**：通过 Git MCP 服务器，理解代码仓库的历史和变更
- **第三方服务连接**：连接 Slack、Google Drive、GitHub 等外部服务
- **开发工具增强**：Zed、Replit、Codeium 等工具通过 MCP 让 AI 代理更好地检索上下文信息

## 二、Skill（技能）

Skill（技能）是 Claude Code 中引入的概念，指的是**一组指令、脚本和资源的集合**，Claude 可以动态加载这些集合以提升在特定任务上的表现。

从本质上看，Skill 是一种**能力封装机制**。它将完成某类任务所需的知识、操作步骤和工具调用逻辑打包在一起，形成一个可复用的模块。当 Claude 遇到匹配的场景时，会自动加载对应的 Skill 来指导其思考和行动。

一个典型的 Skill 结构包括：

- **指令文件**：描述 Skill 的用途、触发条件和执行步骤
- **脚本/工具**：完成具体任务的可执行代码或工具定义
- **资源文件**：辅助理解的文档、模板或参考数据

### 2.2 为什么选择这项技术

Skill 机制解决的是**通用 AI 在专业领域表现不足**的问题。基础大模型虽然具备广泛的通用知识，但在面对特定领域的复杂任务时，往往缺乏：

- 精确的操作流程指导
- 领域特定的术语和约束理解
- 与专用工具的标准化交互方式

通过 Skill 机制，开发者可以将领域专家的知识沉淀为结构化指令，让 AI 在特定场景下表现得更加专业和可靠，而无需对模型本身进行微调。

### 2.3 应用场景

Skill 的典型应用场景包括：

- **代码审查 Skill**：定义代码审查的标准、检查项和输出格式
- **测试编写 Skill**：封装测试框架的使用方法和最佳实践
- **文档生成 Skill**：规范 API 文档、README 的生成格式
- **安全审计 Skill**：定义安全检查清单和漏洞识别流程
- **特定框架开发 Skill**：如 Next.js Skill、Prisma Skill

## 三、Agent（智能体）

### 3.1 是什么

Agent（智能体）是指**能够感知环境、自主决策并执行动作的 AI 系统**。与传统的单次请求-响应式 AI 不同，Agent 具有持续运行、目标导向和自主行动的特征。

Agent 的核心架构通常包含以下组件：

- **感知模块**：接收来自环境的输入（用户指令、系统状态、外部事件等）
- **推理/规划模块**：分析目标、分解任务、制定执行计划
- **行动模块**：调用工具、执行代码、与外部系统交互
- **记忆模块**：维护短期上下文和长期知识，支持持续学习

在 Claude Code 中，Agent 模式允许 Claude 自主执行多步骤任务，如分析代码库、运行测试、修改文件、提交代码等，而无需用户在每一步进行确认。

### 3.2 为什么选择这项技术

Agent 架构解决的是**复杂任务的自动化执行**问题。传统的 AI 交互模式要求用户将任务拆分为多个步骤并逐一指导 AI 执行，存在以下局限：

- 效率低下：复杂任务需要大量人工介入
- 一致性差：人工拆分步骤容易遗漏或出错
- 无法异步执行：AI 不能独立持续工作并汇报结果

Agent 通过赋予 AI 自主规划和执行能力，使其能够独立完成需要多步骤、多工具协作的复杂任务，从而大幅提升自动化水平。

### 3.3 应用场景

Agent 的典型应用场景包括：

- **自动化代码重构**：Agent 自主分析代码库，识别重构点，执行修改并验证结果
- **智能客服**：持续监听用户请求，自主查询知识库、调用业务系统、回复用户
- **数据分析助手**：接收分析需求后，自主获取数据、清洗处理、生成图表、撰写报告
- **运维自动化**：监控系统状态，自主诊断问题、执行修复脚本、上报异常
- **研究助手**：根据研究主题，自主检索资料、整理信息、生成综述文档

## 四、三者的对比

### 4.1 定位对比

| 维度         | MCP                      | Skill                  | Agent             |
| ------------ | ------------------------ | ---------------------- | ----------------- |
| **本质**     | 通信协议/连接标准        | 能力模块/指令集合      | 自主系统/运行实体 |
| **抽象层次** | 基础设施层               | 功能模块层             | 应用系统层        |
| **核心作用** | 定义 AI 与外部的连接方式 | 封装特定领域的执行能力 | 自主完成复杂任务  |
| **交互对象** | 数据源、工具、服务       | AI 模型                | 用户、环境、工具  |

### 4.2 关系与协作

三者在实际系统中通常是**协同工作**的关系：

```
┌─────────────────────────────────────┐
│              Agent（智能体）          │
│  ┌─────────────────────────────┐    │
│  │      Skill（技能）           │    │
│  │  ┌─────────────────────┐    │    │
│  │  │   MCP（协议）        │    │    │
│  │  │  连接外部工具/数据源   │    │    │
│  │  └─────────────────────┘    │    │
│  │  封装领域知识和操作流程         │    │
│  └─────────────────────────────┘    │
│  自主规划、决策、执行多步骤任务          │
└─────────────────────────────────────┘
```

- **MCP 是 Skill 和 Agent 的能力扩展基础设施**：Skill 和 Agent 通过 MCP 协议连接外部世界
- **Skill 是 Agent 的专业能力组件**：Agent 在执行任务时，可以加载不同的 Skill 来获取专业能力
- **Agent 是 Skill 和 MCP 的编排者**：Agent 决定何时调用哪个 Skill，以及通过 MCP 连接哪些外部资源

### 4.3 类比理解

可以用一个**餐厅**的类比来理解三者关系：

- **MCP** 像是餐厅与供应商之间的**标准化采购协议**，规定了食材的规格、交付方式和质量标准。有了这个协议，餐厅可以对接任何符合标准的供应商。
- **Skill** 像是餐厅的**菜谱和烹饪流程**，定义了每道菜的制作步骤、所需食材和成品标准。厨师按照菜谱就能稳定地产出合格的菜品。
- **Agent** 像是**餐厅经理**，接收顾客订单后，自主决定由哪位厨师做什么菜、从哪个供应商采购食材、如何协调出餐顺序，最终完成整个服务流程。

### 4.4 面试常问问题

**Q：MCP 和 API 有什么区别？**

A：MCP 是一种协议标准，而 API 是具体的接口实现。MCP 的目标是统一 AI 与外部系统的连接方式，让 AI 模型能够以一致的方式发现和调用各种工具；API 则是特定系统暴露的功能接口。可以理解为：MCP 是"如何连接"的标准，API 是"连接后能做什么"的具体定义。

**Q：Skill 和 Prompt 有什么区别？**

A：Prompt 是一次性的输入指令，而 Skill 是结构化、可复用的能力模块。Skill 通常包含更丰富的内容（如脚本、资源文件、多步骤流程），并且可以在不同会话中被动态加载和复用。Prompt 更像是临时对话，Skill 更像是培训手册。

**Q：Agent 和传统的 AI 应用有什么区别？**

A：传统 AI 应用通常是"用户问、AI 答"的单轮或多轮对话模式，AI 被动等待用户输入；而 Agent 是**主动**的，它能够自主规划、持续执行、调用工具、处理异常，并在完成目标后向用户汇报。Agent 的核心特征是**自主性**。

**Q：MCP 和 LLM 的 Tool Call / Function Call 有什么区别？**

A：MCP 可以理解为 LLM Tool Call 的**标准化和扩展**：

- **Function Call** 是 LLM 调用预定义函数的机制，工具需要硬编码在应用代码中
- **MCP** 是让 LLM **动态发现和调用**外部工具的标准化协议

关键区别：

- Function Call 的工具集是**有限且预定义**的，MCP 支持**动态发现**可用工具
- Function Call 需要手动定义每个工具，MCP 提供**统一协议**，任何符合规范的服务都能被 AI 调用
- MCP 提供了**工具发现机制**和**统一上下文管理**，跨工具共享状态更便捷
- MCP 将 Function Call 的概念标准化，让不同系统间的 AI 工具调用变得可互操作

## 五、总结

- **MCP** 解决的是"**AI 如何连接外部世界**"的问题，提供标准化的连接协议
- **Skill** 解决的是"**AI 如何具备专业能力**"的问题，封装领域知识和操作流程
- **Agent** 解决的是"**AI 如何自主完成任务**"的问题，实现复杂任务的自动化执行

在实际应用中，三者通常是层层构建的关系：以 MCP 为连接基础，以 Skill 为能力组件，以 Agent 为调度核心，共同构成一个能够感知环境、自主决策、高效执行的 AI 应用系统。
